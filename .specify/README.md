## 1. 📐 スタイルガイド

### 1.1 メタデータ

ファイル先頭にテーブル形式で記載します。

| 項目 | 説明 |
|------|------|
| **機能名** | 機能の名称 |
| **日付** | 作成日または最終更新日（YYYY-MM-DD形式） |
| **ステータス** | 下書き / レビュー中 / 確定 / 完了 など |

### 1.2 目次

- `## 📑 目次` セクションを必ず含める
- H3レベルまでリンクを含める（H4は含めない）
- メタデータの直後、本文の前に配置
- **目次のリンクテキストには絵文字を含めない**（見出しには絵文字があっても、目次では除外する）
  - ✅ 正: `- [コンテンツ品質](#-コンテンツ品質)`
  - ❌ 誤: `- [✅ コンテンツ品質](#-コンテンツ品質)`

### 1.3 セクション見出し

`##` レベルの見出しには必ず絵文字を付けます。

### 2.2 英語維持する特殊文字列

以下の文字列はスクリプトでパースされるため、英語のまま維持します。

#### プレースホルダー・マーカー

- `NEEDS CLARIFICATION` - 要確認マーカー
- `N/A` - 該当なし
- `[P]` - 並列実行可能タスクマーカー
- `[US1]`, `[US2]`, `[US3]` - ユーザーストーリーラベル
- `T001`, `T002`, `T003` - タスクID形式
- `CHK001`, `CHK002` - チェックリストID形式
- `- [ ]`, `- [x]`, `- [X]` - Markdownチェックボックス
- `{{}}` 形式のプレースホルダー

#### ステータス・品質タグ

- `✓ PASS`, `✗ FAIL` - チェックリストステータス
- `Clear`, `Partial`, `Missing` - カバレッジステータス
- `Resolved`, `Deferred`, `Outstanding` - 解決ステータス
- `[Gap]`, `[Ambiguity]`, `[Conflict]`, `[Assumption]` - 品質タグ
- `[Completeness]`, `[Clarity]`, `[Consistency]`, `[Coverage]` - 品質次元
- `[Spec §X.Y]` - 仕様参照形式
- `[RESOLVED]`, `[COVERED]`, `[WONT_DO]` - チェックリストレビューマーカー

#### ファイル名・技術用語

- ファイル名: `spec.md`, `plan.md`, `tasks.md`, `research.md`, `data-model.md`, `quickstart.md`
- 言語名: `Python`, `Rust`, `JavaScript`, `TypeScript`, `Go`, `Ruby`, `PHP`, `Java`, `C#`, `Swift`, `Kotlin`, `C`, `C++`, `R`
- フレームワーク名: `FastAPI`, `React`, `Node.js`, `Django` 等

#### 優先度・重要度

- `P1`, `P2`, `P3` - 優先度レベル
- `CRITICAL`, `HIGH`, `MEDIUM`, `LOW` - 重要度レベル
- `MAJOR`, `MINOR`, `PATCH` - バージョニング

### 2.3 翻訳ルール

1. **技術用語は原則カタカナ表記**: 例: Storage → ストレージ, Framework → フレームワーク
2. **一般的な英語は日本語に**: 例: Requirements → 要件, Success → 成功
3. **略語は維持**: 例: MVP, TDD, API, CLI
4. **固有名詞は英語維持**: 例: GitHub, Copilot, Claude
5. **コードブロック内は英語維持**: パス、コマンド、コード例

## 3. � 表記ガイド

### 3.1 フェーズ表記

tasks.mdでは `## 🚀 3. Phase` の配下に各フェーズを `###` サブセクションとして配置します。

| 表記 | 用途 | 例 |
|------|------|-----|
| `3.1:` | セットアップフェーズ | `### 3.1: セットアップ（共有インフラストラクチャ）` |
| `3.2:` | 基盤フェーズ | `### 3.2: 基盤（ブロッキング前提条件）` |
| `3.3:`〜`3.7:` | ユーザーストーリーフェーズ | `### 3.3: US1 - [タイトル] [P1]` |
| `3.8:` | 仕上げフェーズ | `### 3.8: 仕上げとクロスカッティング関心事` |

### 3.2 ユーザーストーリーID（US）

| ID | spec.md参照 | tasks.md参照 | 説明 |
|----|-------------|--------------|------|
| `US1` | §2.1 | `[US1]` | 最初のユーザーストーリー（通常P1/MVP） |
| `US2` | §2.2 | `[US2]` | 2番目のユーザーストーリー（通常P2） |
| `US3` | §2.3 | `[US3]` | 3番目のユーザーストーリー（通常P3） |
| `US4`〜 | §2.4〜 | `[US4]`〜 | 追加のユーザーストーリー |

**spec.mdでの使用**:
```markdown
### 2.1 US1: タスクを作成できる [P1]
### 2.2 US2: タスクを完了できる [P2]
```

**tasks.mdでの使用**:
```markdown
## 📝 Phase3: US1 - タスクを作成できる [P1]
- [ ] T012 [P] [US1] src/models/task.pyにTaskモデルを作成
```

### 3.3 優先度表記

| 優先度 | 意味 | 説明 |
|--------|------|------|
| `P1` | 必須 | MVPに必須。これがないとリリースできない |
| `P2` | 重要 | MVPに含めたい。時間が許せば実装 |
| `P3` | あれば良い | 将来の拡張として検討。今回は実装しない可能性あり |

**見出しでの使用**: `[P1]`、`[P2]`、`[P3]` を見出し末尾に付与
```markdown
### 2.1 US1: タスクを作成できる [P1]
```

## 5. 🔄 本家アップデート時の対応

本家SpecKitがアップデートされた場合：

1. 新しい用語が追加された場合、用語対応表に日本語訳を追加
2. 既存の用語が変更された場合、対応する日本語訳も更新
3. 変更履歴セクションに更新内容を記録
4. 関連するテンプレート・エージェントファイルも同時に更新
